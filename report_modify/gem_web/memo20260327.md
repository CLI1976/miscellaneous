# 2026-03-27 開發紀錄：Gemini Web 自動化報告修改

## 目標

將原本的 Gemini API 模式（`gem.py` + `Gem_report.ahk`）改為 Playwright 控制 Gemini Web 版，主要動機：
- 使用自訂 Gem（已內建 system instruction，省去 prompt 撰寫）
- 支援二次對話修改
- 省去 API 費用
- 實際使用速度與 API 版本無差異

---

## 過程與踩坑紀錄

### 1. Google 登入被擋（自動化瀏覽器偵測）

**現象**：Playwright 啟動 Chrome（`channel="chrome"` 或 Playwright Chromium）嘗試登入 Google，出現「這個瀏覽器或應用程式可能有安全疑慮」。

**根本原因**：Google 登入頁面會偵測 `--enable-automation` flag（Playwright 預設加入）及 `navigator.webdriver` 屬性。

**嘗試過的方案**：
- `channel="chrome"` → 仍被擋
- `launch_persistent_context` + 真實 Chrome profile（預設路徑）→ Chrome 報錯：`DevTools remote debugging requires a non-default data directory`
- `launch_persistent_context` + `C:\Temp\chrome-debug` + `ignore_default_args=["--enable-automation"]` → Chrome 開啟但 Gemini session 不穩定，3 小時內 session 強制失效

**最終解法**：不讓 Playwright 啟動 Chrome，改用 CDP connect。

---

### 2. CDP 連接（`--remote-debugging-port=9222`）無法開啟

**現象**：Chrome 加上 `--remote-debugging-port=9222` 後，`netstat` 顯示 port 未開啟。

**根本原因**：Chrome 新版安全政策：使用預設 user data dir 時靜默忽略 `--remote-debugging-port`。

**解法**：加上 `--user-data-dir="C:\Temp\chrome-debug"`（非預設路徑），port 9222 才會正常開啟。

---

### 3. Selector 問題

**送出按鈕**：英文 aria-label `"Send message"` 無效，Gemini 中文介面實際為 `"傳送訊息"`。

**輸入框**：`rich-textarea div[contenteditable="true"]` 匹配到兩個元素（Quill editor 有 `.ql-editor` 和 `.ql-clipboard` 兩個 contenteditable div），strict mode 報錯。改用 `rich-textarea div[role="textbox"]` 唯一匹配。

---

### 4. 貼上文字無效（ClipboardEvent isTrusted）

**現象**：JavaScript 的 `ClipboardEvent` 貼上回傳成功，但 Gemini 輸入框實際上是空的。

**根本原因**：Quill editor 只接受 `isTrusted: true` 的剪貼簿事件（真實鍵盤操作），JavaScript 合成的事件被忽略。

**解法**：改用 PowerShell `Set-Clipboard` 設定系統剪貼簿，再由 Playwright 執行 `Ctrl+V`，產生真實的受信任事件。

```python
subprocess.run(
    ["powershell", "-NonInteractive", "-Command",
     f'Get-Content -Path "{tmp_clip}" -Encoding UTF8 -Raw | Set-Clipboard'],
    ...
)
await page.keyboard.press("Control+v")
```

---

### 5. Session 快速失效（3 小時內）

**現象**：`storageState.json` 方式（把 Chrome 的 cookie 存成 JSON，載入 Playwright Chromium）跑了幾次後 Gemini session 失效。

**根本原因**：Google 偵測到同一帳號從不同瀏覽器指紋（Playwright Chromium）登入，判定為異常並強制 session 失效。

**最終解法**：CDP connect 到使用者自己啟動的 Chrome（無 automation flags），Google 視為正常瀏覽器，session 長期穩定。

---

## 最終架構

```
gem_setup.py（一次性）
  用戶手動啟動 Chrome：chrome.exe --remote-debugging-port=9222 --user-data-dir="C:\Temp\chrome-debug"
  → 登入 Google
  → gem_setup.py 透過 CDP 擷取 storageState（備用）

gem_web.py（日常使用）
  → CDP connect_over_cdp("http://localhost:9222")
     - 若 Chrome 未執行，自動 subprocess.Popen 啟動
  → contexts[0].new_page()（使用既有登入 session）
  → goto Gem URL
  → PowerShell Set-Clipboard + Ctrl+V 貼上
  → 點擊「傳送訊息」
  → 等待送出按鈕 disabled → enabled（回應完成判斷）
  → 寫 gem_done.txt 通知 AHK
  → page 保持開啟

Gem_report.ahk（F2 熱鍵）
  → UIA 讀取 Notepad++ 文字
  → 寫入 gem_in.txt
  → shell.Run gem_web.py（隱藏視窗）
  → 輪詢 gem_done.txt → 顯示完成 Tooltip
```

## 關鍵技術結論

| 問題 | 錯誤方向 | 正確方向 |
|------|---------|---------|
| Google 登入偵測 | 用各種方式繞過偵測 | 根本不讓 Playwright 登入，用已登入的 Chrome |
| Session 失效 | 定期重存 storageState | CDP connect 維持同一 Chrome session |
| 貼上亂碼/無效 | JavaScript ClipboardEvent | 系統剪貼簿 + 真實鍵盤 Ctrl+V |
| Selector 不對 | 依賴英文 aria-label | 在瀏覽器查實際 aria-label（中文介面） |
